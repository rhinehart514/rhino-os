# Release Notes Template

## Structure

```markdown
## [Version]: [Thesis summary — what this release proves]

[2-3 sentences: what changed for the user. Not what code changed — what's different for them.]

### What's new
- [User-facing change — specific, measurable if possible]
- [User-facing change]
- [User-facing change]

### What we learned
- [Key insight from this version's work — optional, include if genuinely interesting]

### Known limitations
- [Honest gap — what's not done yet]

### What's next
[One sentence: what the next version will prove]
```

## Rules

- **User-facing only.** "Refactored the scoring pipeline" is not a release note. "Scores are now 3x faster" is.
- **Trace to evidence.** Every bullet should map to a proven roadmap evidence item or a measurable change.
- **No slop.** Same ban list as all copy: no revolutionary, cutting-edge, seamlessly, etc.
- **Honest limitations.** If something isn't done, say so. Users respect honesty more than spin.
- **One theme per release.** If the release does 5 things, find the one narrative that ties them together.

## Anti-patterns

- Listing every file changed — nobody cares
- "Bug fixes and improvements" — this means nothing. Name the bugs, name the improvements.
- Apologizing for limitations — state them matter-of-factly
- Promising timelines for future work — you don't know. Say what's next, not when.
