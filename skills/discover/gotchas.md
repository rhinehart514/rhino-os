# Discovery Gotchas

Real failure modes from running /discover. Read before starting.

- **Spec without founder input**: Agents research, but the FOUNDER decides. Never generate a spec purely from agent output. The founder's answers are the spec; agent research validates and enriches.

- **Vague who accepted**: The #1 failure. "Developers" is not a who. "Solo technical founders in month 3 with code but no users" is. Push back EVERY time the who is a category.

- **Wiring without approval**: Never auto-wire (create features, assertions, roadmap) before the founder approves the spec. The spec is a contract — get sign-off.

- **All questions at once**: Don't dump 9 questions in one message. One question at a time via AskUserQuestion. React to answers. Follow threads. The conversation IS the discovery.

- **Skipping market-2026 context**: Every session must read references/market-2026.md. "Why now?" without a 2026 signal is not a valid answer.

- **Thin kill list**: Founders resist killing ideas. A spec with 2 items in not_building hasn't been thought through. Push for 5+.

- **Soft pivot triggers**: "If users don't come back" needs a number. How many users? After how many days? Specificity is the difference between a trigger and a wish.

- **Refine without evidence**: Attacking a spec is useless without data. Read eval-cache, predictions, assertions BEFORE entering refine mode. Attack with evidence, not opinion.

- **Systems-first thinking**: Designing perfect systems before understanding if anyone wants them. Start from the user in define mode. Systems mode is for AFTER you know the user.

- **Analysis paralysis**: Discovering is comfortable. Building is uncomfortable. The spec should be "good enough to wire" not "perfect." Quality gate is 70/100, not 100/100.

- **Duplicate tasks**: Always read existing todos.yml before generating tasks. Duplicate tasks are noise.

- **Scope creep through discovery**: Every session finds "just one more thing." Scope expands. The kill list exists to fight this.
