Scoring system

Two tiers. Training loss runs every commit (cheap, grep-based). Eval loss runs on demand (expensive, visual). Track dimensions independently. No composites. The trend is the signal.

The ML analogy — why two tiers?

In ML, training loss and eval loss measure different things. Training loss is cheap and frequent. Eval loss is expensive but real. You need both.

score.sh

grep-based, every commit, 5 dimensions
= training loss
vs

taste.mjs

screenshots + Claude vision, on demand, 8 dimensions
= eval loss
why not just use taste?

Taste costs ~$0.50 per run and takes 30 seconds. You can't run it on every commit. score.sh costs nothing and runs in 2 seconds. It catches structural regressions instantly. Taste catches the things grep can't see — but it runs when you choose to look.

Visual evaluation. Screenshots every route, sends to Claude vision, scores what it sees.

Training loss — 5 dimensions (score.sh)

Each dimension tracked independently. No weighted average — composites hide problems. When score.sh returns a single number, it returns the LOWEST dimension (weakest link).

=== rhino lint ===

  ✓ build gate: PASS (100/100)

  structure:     100/100  ·  dead ends, empty states
  product:        32/100  ·  creation→share, return pull, social
  capabilities:    0/100  ·  routes, components, auth, depth
  hygiene:        95/100  ·  hardcoded colors, any types, console.log

  taste:          —  run: rhino taste eval

  Project: nextjs (apps/web/src)
  History: .claude/scores/history.tsv (12 runs)

Build health gate

Does it compile? This is pass/fail. If build fails, nothing else matters.

Checks for TypeScript errors (with --force) or build artifact freshness. Stale builds (>24h) get penalized. No artifacts = -30.

Structure subtractive

Dead-end pages (no links/buttons out), empty states without CTAs. Starts at 100, subtracts problems.

Finds all page.tsx files. Checks each for navigation elements (Link, href, onClick). Dead ends and CTA-less empty states reduce the score proportionally.

Product flows additive + depth

Measures FLOW COMPLETION, not feature existence. "Can a user create → save → share → someone sees it?"

Four sub-flows: creation→distribution (0-30), return pull (0-30), social (0-20), real user signals (0-20). Each step requires 2+ file matches to avoid false positives. One accidental keyword match doesn't count.

Capabilities additive

How many user-facing features exist at real depth? Routes, components, API endpoints, auth, search, media, a11y.

Calibrated so a medium app scores 40-60. Needs 26+ pages for max route score, 101+ components for max UI score. Auth requires 3+ files (real integration, not one import).

Code hygiene subtractive

Hardcoded hex colors, : any types, console.log in production, TODO/FIXME/HACK comments.

Starts at 100. Thresholds: >20 hardcoded colors = -25, >20 any types = -25, >20 console.logs = -20, >20 TODOs = -15. Excludes node_modules and .d.ts.
no composite score

Build=100 + product=0 averaging to 50 hides a critical gap. Each dimension is tracked independently in history.tsv. When scripts need a single number, they get the LOWEST dimension — surfacing the weakest link, not a comfortable average.

Eval loss — 8 taste dimensions (taste.mjs)

Score based on what you SEE, not what you imagine the code does. Each dimension 1-5. Compared against products users actually use — Instagram, Discord, Notion, Linear.

Hierarchy

5 = clear visual order, primary action dominates
1 = everything same size/weight, no priority

Breathing room

5 = intentional whitespace, spacious rhythm
1 = cramped, no padding, walls of content

Contrast

5 = primary actions unmissable, clear interactive distinction
1 = everything blends, can't tell what's clickable

Polish

5 = transitions, loading feedback, micro-animations
1 = dead clicks, jarring state changes

Emotional tone

5 = recognizable personality, feels like THIS product
1 = mismatch (creative tool looks corporate)

Information density

5 = goldilocks — useful, scannable, not overwhelming
1 = wastefully empty or wall of content

Wayfinding

5 = next action always obvious, no dead ends
1 = users would get lost, no clear path

Distinctiveness

5 = recognizable in a lineup, beyond framework defaults
1 = pure template, looks like every other app
calibration

A score of 3 is "fine." Most AI-generated UIs score 2-3 across the board — that's the starting point, not failure. 5 means "this specific thing was clearly thought about." Evidence must be cited: "the hero text competes with sidebar nav for attention." Compared against products users actually use daily (Instagram, Discord, Notion, Linear).

How scores drive decisions

Scores aren't reports. They feed forward into the next build cycle.

1. Score after build

rhino score . runs after implementation. Each dimension compared to previous run. Trend arrows show direction.

2. Strategy reads scores

rhino strategy reads history.tsv and identifies the weakest dimension. Next sprint targets the bottleneck.

3. Experiment loop

Baseline score → change → re-score → keep if improved, discard if not. The score is the experiment's success criterion.

Product eval — the ship gate (0-1 scale)

Separate from both tiers above. /product-eval asks: "would anyone actually use this?" Scores 4 mandatory ceiling dimensions on a 0-1 scale.

Escape velocity

Is there ONE person who needs this TODAY? Not "could be useful" — needs.

UI/UX uniqueness

Would a user recognize this in a lineup? Or does it look like every other AI-generated app?

IA benefit

Does this product get BETTER because AI built it? Or is AI just a faster keyboard?

Return pull

Day 3: would they come back? Day 7? This is the hardest dimension to score and the most important.

three scales, one purpose

score.sh (0-100, per dimension) = structural health. Run constantly.
taste.mjs (1-5 → 0-100) = visual quality. Run on demand.
/product-eval (0-1) = ship readiness. Run before launch.

All three feed forward: gaps from product-eval become sprint priorities, which get built and measured by score.sh, then visually verified by taste.
