# Product Thinking Reference

Read on demand when running product lenses. Not loaded into SKILL.md.

## The 5 Value Questions

Ask these before every feature, every idea, every session. If you can't answer them, you're building in the dark.

1. **Who gets value?** Name the human. "Users" is not a name. "A solo founder who just cloned this and wants their project to get better in one session" is.
2. **What changes for them?** After they use this, what's measurably different? If nothing changes, it's not value.
3. **How fast?** Time from "I found this" to "I got value." Every minute is a chance to lose them. Target: value in the first session, ideally first 5 minutes.
4. **Would they notice if it disappeared?** If you removed it tomorrow, would anyone complain? No complaints = furniture, not value.
5. **What's the return trigger?** Why would they come back tomorrow? No pull = no retention.

## Stage-Appropriate Expectations

| Stage | Score range | Focus | Acceptable debt | Red flags |
|-------|------------|-------|-----------------|-----------|
| Pre-product | N/A | Name the person, test demand | Everything | Building before validating demand |
| Stage one (0 users) | 30-60 | First loop: discover → value → return | Low craft, rough edges, manual processes | Polishing, growth features, pricing optimization |
| Stage some (1-10) | 50-75 | Retention: do they come back? Why not? | Feature gaps, imperfect onboarding | New features instead of retention fixes |
| Stage many (10-100) | 65-85 | Distribution: how do users find this? | Minor UX, incomplete docs | Building without distribution plan |
| Growth (100+) | 75-95+ | Unit economics: does the business work? | Edge case gaps | Revenue avoidance, ignoring churn |

## When to Pivot

Pivot signals (any 2 = seriously consider it):
- Value hypothesis has been tested and failed
- The named person doesn't exist or doesn't care
- 3+ assumptions disproven in a row
- Building for 4+ weeks with no score movement
- Every idea feels incremental — no excitement about the direction

Pivot is NOT:
- Changing a feature (that's iteration)
- Adding a new user segment (that's expansion)
- Rewriting architecture (that's refactoring)

Pivot IS:
- Changing WHO you serve
- Changing WHAT problem you solve
- Changing HOW you deliver value

## The Assumption Stack

Every product is a stack of assumptions. From most dangerous to least:

1. **Demand**: "This person has this problem" — if wrong, nothing else matters
2. **Solution**: "This approach solves it" — the mechanism might be wrong
3. **Mechanism**: "They'd use it this way" — the form factor might be wrong
4. **Market**: "Nothing else does this" — competitors might already solve it
5. **Timing**: "This is needed now" — why now, not last year?
6. **Business**: "This can sustain" — revenue model exists

Test top-down. Don't optimize the business model for a product nobody wants.
